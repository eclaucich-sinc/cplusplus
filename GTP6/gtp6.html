Ejercicio 1

NoteConsigna

Refactorizar la simulación de campeonato playoffs compuesto de objetos abstractos player_t del Tic-Tac-Toe implementando una gestión de ciclo de vida seguro de memoria a través de std::unique_ptr.

El uso de unique_ptr garantiza que exista un único propietario autorizado sobre la memoria asignada de un objeto dinámico. Cuando un objeto player_t queda fuera del torneo y ya no se utiliza, este puntero inteligente permite liberar la memoria automáticamente al salir del ciclo sin depender explícitamente de la directiva manual delete, eliminando potenciales memory leaks.

Dado que un puntero de este tipo es estricto e in-copiable, los jugadores que avanzan de ronda deben ser reubicados a la siguiente estructura moviendo el puntero en sí mediante std::move:

void playoffs(vector<unique_ptr<player_t>> players) {
    while(players.size() > 1) {
        vector<unique_ptr<player_t>> next_round;
        
        for (size_t i = 0; i < players.size(); i += 2) {
            // ... (Se simulan multiples partidos entre players[i] y players[i+1]) ...
            
            // Reasignación del ganador a la siguiente ronda
            if (p1_wins > p2_wins) {
                next_round.push_back(std::move(players[i]));
            } else {
                next_round.push_back(std::move(players[i+1]));
            }
        }
        
        // Al sobreescribir la capa base actual, todos los punteros abandonados 
        // (perdedores) se autodestruyen liberando correctamente su memoria.
        players = std::move(next_round);
    }
}

Ejercicio 2

NoteConsigna

Implementar las funciones de figuras geométricas (geoshape_t) de los GTPs anteriores, asegurando de forma estricta el ciclo de vida de los objetos compuestos (union_t, intersection_t, difference_t) mediante std::unique_ptr. Validar relaciones lógicas numéricas con áreas compuestas.

Al componer figuras, los objetos que envuelven a múltiples sub-figuras se convierten en el propietario exclusivo de su memoria. Para simplificar la lectura y escritura de la jerarquía, se definió un alias sobre el puntero inteligente:

typedef unique_ptr<geoshape_t> gsp_t;

Este alias es utilizado por las clases contenedoras (como union_t, intersection_t y difference_t) para alojar la estructura interna de sub-figuras, garantizando que éstas sean destruidas automáticamente cuando el contendor principal deje de existir. A continuación se observa su implementación en la intersección:

class intersection_t : public geoshape_t {
public:
    vector<gsp_t> shapes; // Vector de punteros de propiedad única

    bool inside(const vector<double> &x) override {
        if (shapes.empty()) return false;
        for (const auto& shape : shapes) {
            if (!shape->inside(x)) return false; // Llama recursivamente a inside()
        }
        return true;
    }
    // ...
};

En el bucle principal, aprovechar make_unique al inyectar figuras en este arreglo shapes protege la aplicación en caso de una falla o excepción durante la cadena de inicialización. Para comprobar el correcto funcionamiento de toda esta estructura de punteros polimórficos, se calculó por integración numérica la propiedad topológica disjunta de conjuntos para dos círculos cualesquiera: \(A \cup B = (A-B) \cup (B-A) \cup (A \cap B)\).

Ejercicio 3

NoteConsigna

Desarrollar un procesador geométrico capaz de instanciar arreglos de objetos directamente leyendo desde un formato serializado .json. Empleando RTTI y punteros de propiedad única unique_ptr, generar reportes para todas las combinaciones generables sin duplicación de recursos y guardar un output.json.

Primero necesitamos leer el archivo y extraer de este el tipo de figura y sus parámetros. Esto se conoce como parsear el json y nos permitirá crear las figuras apropiadas según se establezcan en el archivo de input y las operaciones correspondientes:

gsp_t parse_shape(const json& j) {
    if (!j.contains("type")) return nullptr;
    string type = j["type"];
    if (type == "circle") {
        return make_unique<circle_t>(j["xc"], j["yc"], j["R"]);
    } else if (type == "square") {
        double xc = j["xc"];
        double yc = j["yc"];
        double L = j["L"];
        return make_unique<square_t>(xc - L/2.0, yc - L/2.0, xc + L/2.0, yc + L/2.0);
    } else if (type == "ellipse") {
        return make_unique<ellipse_t>(j["cx"], j["cy"], j["rx"], j["ry"]);
    } else if (type == "convex_polygon") {
        vector<double> pts = j["pts"].get<vector<double>>();
        return make_unique<convex_polygon_t>(pts);
    } else if (type == "union") {
        auto u = make_unique<union_t>();
        if (j.contains("A")) u->shapes.push_back(parse_shape(j["A"]));
        if (j.contains("B")) u->shapes.push_back(parse_shape(j["B"]));
        return u;
    } else if (type == "intersection") {
        auto i = make_unique<intersection_t>();
        if (j.contains("A")) i->shapes.push_back(parse_shape(j["A"]));
        if (j.contains("B")) i->shapes.push_back(parse_shape(j["B"]));
        return i;
    } else if (type == "difference") {
        auto d = make_unique<difference_t>();
        if (j.contains("A")) d->shapes.push_back(parse_shape(j["A"]));
        if (j.contains("B")) d->shapes.push_back(parse_shape(j["B"]));
        return d;
    }
    return nullptr;
}

Es crucial resaltar que para construir pares dinámicos dentro de estructuras operativas como union_t utilizando un unique_ptr, es un requisito estricto trasladar permanentemente la titularidad del puntero inteligente hacia el objeto que calculará su área transitoria, y devolverla al array principal en cuanto el objeto agote su vida útil.

// Función de comparación para pares 'i' y 'j'
union_t uni;

uni.shapes.push_back(std::move(shapes[j])); // Traspaso de la propiedad a la Unión

double uniAB = area(uni, N); // El área interna invoca los shapes movidos

shapes[i] = std::move(uni.shapes[0]); // Devolución de titularidad a 'shapes'
shapes[j] = std::move(uni.shapes[1]); // Devolución de titularidad a 'shapes'

La función principal checkgeo primero se encargará de realizar el parse del json e itera sobre cada par de figuras encontradas, realizando las operaciones correspondientes y guardando los resultados en un archivo json:

json checkgeo(json &in, int N=100) {
    vector<gsp_t> shapes;
    for (auto& item : in) {
        shapes.push_back(parse_shape(item));
    }

    json out = json::array();
    
    int n = shapes.size();
    for (int i = 0; i < n; ++i) {
        for (int j = i + 1; j < n; ++j) {
            json pair_res;
            pair_res["idA"] = i;
            pair_res["idB"] = j;
            pair_res["typeA"] = typeid(*shapes[i]).name();
            pair_res["typeB"] = typeid(*shapes[j]).name();
            
            double a_val = area(*shapes[i], N);
            double b_val = area(*shapes[j], N);
            pair_res["A"] = a_val;
            pair_res["B"] = b_val;
            //...
        }
    //...
    }
    
    return out;
}

Ejercicio 4

NoteConsigna

Reescribir el análisis y guardado JSON del ejercicio anterior remplazando la directiva de memoria de pertenencia estricta por un sistema multi-pertenencia compartida: std::shared_ptr.

A diferencia de unique_ptr, el uso de shared_ptr permite simplificar ampliamente la arquitectura geométrica que requiera de entrelazar repetidamente elementos a lo largo de diversas combinaciones. Mediante la incorporación nativa en memoria de un “bloque de control”, este tipo de puntero es capaz de contabilizar la cantidad de referencias vivas instanciadas de manera transparente, permitiendo la clonación local o superficial del puntero sin copiar los costosos objetos lógicos del interior.

De esta forma, ya no tenemos la necesidad de usar la semántica destructiva std::move() al inyectar punteros al objeto envolvente union_t:

typedef shared_ptr<geoshape_t> gsp_t;

// ...

// Formulación relacional sin transferencias manuales
union_t uni;
uni.shapes.push_back(shapes[i]); // Referencia compartida
uni.shapes.push_back(shapes[j]); // Referencia compartida
double uniAB = area(uni, N);

// Una vez destruido `uni`, su bloque de control reduce las referencias en -1.
// La figura original sobrevive intacta dentro de shapes[]. No hubo fugas.

La salida obtenida es exactamente igual al caso anterior, pero su implementacion es mucho mas limpia y elegante, ademas de que se reduce la cantidad de codigo innecesario.
